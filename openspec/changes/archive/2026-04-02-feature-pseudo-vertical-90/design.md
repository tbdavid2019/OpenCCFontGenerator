## Context

該功能的目的是對已完成 OpenCC 轉換的 TTF 字型進行幾何後處理。傳統的 `vert` 功能由渲染引擎控制旋轉，但「偽直排」則將旋轉直接寫入字型數據。

## Goals / Non-Goals

**Goals:**
- 提供一種方式將所有字元順時針旋轉 90 度。
- 旋轉應以字元的幾何中心為軸（預設為 500, 500）。
- 確保旋轉後字元在水平基線上對齊良好。

**Non-Goals:**
- 不進行 WOFF2 格式處理（僅限 TTF）。
- 不處理 OpenType 佈局功能中的旋轉（如 `vkrn`），僅處理靜態幾何數據。

## Decisions

### 1. 旋轉演算法與中心點
- **選擇**：以 (500, 500) 為中心進行旋轉，適用於大多數 1000-UPM 的中文字型。
- **計算式**：`x' = (y - 500) + 500`, `y' = -(x - 500) + 500`。
- **理由**：相對於以 (0,0) 為中心，以幾何中心旋轉可以減少大規模的位移調整，且排版一致性較高。

### 2. 複合字元 (Composite Glyphs) 處理
- **選擇**：對所有組件 (components) 應用的 `transform` 矩陣進行旋轉調整，或將其「打散 (flatten)」為輪廓再旋轉。
- **決策**：優先嘗試調整組件的偏移量。若複雜度過高，則建議先對 `glyf` 數據進行標準化處理。

### 3. 字型度量 (Metrics) 調整
- **決策**：所有字元的 `advanceWidth` 設定為固定值（例如 1000），模擬直排時字元垂直等距的效果。

### 4. 互動設計 (Interactive Wizard)
- **選擇**：仿照 `start.py` 的架構建立 `start90.py`。
- **流程**：
  1. 輸入字型路徑 (預設可接收主程式輸出的 `_TC.ttf`)。
  2. 設定旋轉中心點 (預設 500, 500)。
  3. 設定是否也旋轉非漢字字元。
  4. 確認執行。
- **目標**：讓使用者無需下命令列參數即可完成旋轉。

## Risks / Trade-offs

- **字距問題 (Kerning)**：原有的水平字距調整將失效。由於是偽直排，使用者通常會接受等寬排版。
- **記憶體消耗**：處理大型中文字型 JSON 時會佔用大量 RAM，建議在處理時採用串流或分塊讀取（若 `otfcc` 支援）。
